# Deploying Aeruvo

This app deploys to **Vercel** (via Nitro's `vercel` preset, pinned
explicitly in `vite.config.ts`). `npm run build` always produces the same
output regardless of where it's run:

```
.vercel/output/static/      → static assets (icons, manifest, JS/CSS, sw.js)
.vercel/output/functions/   → the SSR serverless function (__server.func)
.vercel/output/config.json  → Vercel Build Output API v3 routing config
```

This is Vercel's [Build Output API](https://vercel.com/docs/build-output-api/v3)
format — the same format Vercel's own framework presets (Next.js, etc.)
produce internally. Vercel auto-detects `.vercel/output/` and deploys it
directly, **no `vercel.json` required**.

## Why the app previously 404'd on Vercel

The Nitro preset was pinned to `cloudflare-module`, which writes
Cloudflare-Workers-shaped output (`dist/client` + `dist/server` +
`wrangler.json`) — a format Vercel doesn't understand. Vercel's build
completed without error (it just ran `vite build`), but had no routing
config telling it how to serve `/`, so every request fell through to a
404. Switching the preset to `vercel` fixes this at the source: Nitro now
emits the exact directory structure and `config.json` routing rules Vercel
expects.

## How requests are routed (from `config.json`)

```
1. /assets/*  → served as static files with long-lived immutable cache headers
2. (filesystem) → any other static file that exists (icons, manifest, sw.js) is served directly
3. /*  → falls through to the __server serverless function for SSR
```

There is no static `index.html` at the output root — `/` is intentionally
handled by the SSR function, which renders the full page server-side on
every request. This was verified by directly invoking the built function
locally and confirming it returns `200` with the real rendered HTML
(`<title>Aeruvo — What to wear today</title>`), not just by checking that
files exist on disk.

## Option A — Deploy via the Vercel dashboard (recommended)

1. Push this repo to GitHub/GitLab/Bitbucket.
2. In Vercel, **Add New Project** → import the repo.
3. Vercel will detect there's no framework-specific preset needed since
   `.vercel/output` is generated by the build — set:
   - **Build Command:** `npm run build`
   - **Output Directory:** leave as default (Vercel reads `.vercel/output`
     automatically when present; this field is ignored for Build Output
     API v3 projects)
   - **Install Command:** `npm install`
4. Add your environment variables (see below) in Project Settings →
   Environment Variables before the first build.
5. Deploy. Connect `aeruvo.app` under Project Settings → Domains.

## Option B — Deploy via the Vercel CLI

```
npm install -g vercel        # or use npx vercel each time
vercel login
npm run build
vercel deploy --prebuilt --prod
```

`--prebuilt` tells Vercel to use the `.vercel/output` directory that's
already on disk instead of re-running its own build — useful for
confirming locally exactly what will ship before deploying.

### Smoke-testing the build before deploying

```
npm run build
npx vercel dev
```

or, to test the actual built function directly without the Vercel CLI:

```js
// node smoke-test.mjs
import handler from "./.vercel/output/functions/__server.func/index.mjs";
const res = await handler.fetch(new Request("http://localhost/"));
console.log(res.status, (await res.text()).slice(0, 120));
```

This invokes the same Web-standard `fetch`-shaped handler Vercel's Node.js
runtime calls in production — it doesn't go through the CLI or any dev
server, so it's the most direct way to confirm the build is correct before
deploying.

## Environment variables in production

Every `VITE_*` variable (see `.env.example`) is inlined into the client
bundle **at build time**, not read at runtime. That means:

- For local development: put them in a `.env` file (gitignored, never
  commit it).
- For Vercel: set them in Project Settings → Environment Variables. Vercel
  injects them automatically during its build step (or your local
  `npm run build` if you deploy with `--prebuilt`), so no extra
  configuration is needed beyond entering the values.
- The only genuinely server-side secret this app would ever need is a
  Stripe **secret** key, and only once `/api/create-checkout-session` (see
  `src/lib/billing.ts`) is actually implemented as a server route — that
  one should be added as a Vercel environment variable **without** the
  `VITE_` prefix (so it's never inlined into the public client bundle) and
  read only from server-side code.

## Switching back to Cloudflare later

If you ever need to deploy to Cloudflare Workers again, change the
`nitro.preset` value in `vite.config.ts` back to `"cloudflare-module"` and
restore the Cloudflare-specific `output`/`cloudflare` options that were
removed when switching to Vercel:

```ts
nitro: {
  preset: "cloudflare-module",
  output: { dir: "dist", serverDir: "dist/server", publicDir: "dist/client" },
  cloudflare: { nodeCompat: true, deployConfig: true },
},
```

Then follow the Wrangler-based deploy steps (`wrangler deploy --config
dist/server/wrangler.json`) instead of the Vercel steps above.
